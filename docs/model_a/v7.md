## Hipótesis

La inspección del CSV crudo en v6 identificó que los FP son formularios con nombres y contraseñas españolas que usan encoding Latin-1 (`%F1`=ñ, `%ED`=í, `%FA`=ú). Se esperaba que los ataques no tuvieran Latin-1 — y que esa asimetría fuera la señal faltante.

**Features candidatas:**
- `content_pct_latin1_density` = proporción de `%[CcDdEeFf][0-9A-Fa-f]` en el body
- `url_pct_latin1_density` = proporción de `%[CcDdEeFf][0-9A-Fa-f]` en la URL

### Por qué falló la hipótesis

Los ataques de CSIC 2010 se generaron contra una tienda española. El generador de ataques construye requests con parámetros españoles en los **nombres de campo** — aunque el *valor* del parámetro contenga el payload de inyección, el resto del body tiene caracteres acentuados al mismo ritmo que un request normal:

```
# Ataque POST — el campo "apellidos" tiene Latin-1, el valor "pass" tiene la inyección
apellidos=Garc%EDa&pass=%27+OR+1%3D1--&email=test%40test.com
```

El generador no distingue entre qué parte del request es el payload y qué parte es contexto. Por eso la distribución de Latin-1 en POST es prácticamente idéntica entre normal y ataque (media 0.00420 vs 0.00413).

Para `url_pct_latin1_density` la correlación es **positiva** en GET (+0.131): los ataques GET tienen *más* Latin-1 en la URL que el tráfico normal — en dirección opuesta a lo esperado. Los ataques de tipo parameter tampering incluyen valores con caracteres españoles en la query string.

## Análisis — límite del enfoque actual

### ROC-AUC sigue mejorando

LightGBM v7: ROC-AUC 0.9677 (+0.0016 vs v6). La capacidad de separación del modelo aumentó. Pero esa mejora no se traduce en menos FP al threshold actual porque el modelo no puede encontrar un punto de corte que mejore simultáneamente Recall y Precision.

### Los 936 FP son estructuralmente ambiguos

Después de v5, v6 y v7, el patrón es claro: los FP restantes son requests normales que el modelo no puede diferenciar de ataques con las dimensiones disponibles:

| Dimensión analizada | Resultado |
|---|---|
| Longitud y estructura de URL/body | Explorado desde baseline — señal agotada |
| Indicadores de keywords (`%27`, `SELECT`) | 98.6% de FP sin ninguno |
| Estructura de query string (`url_path_depth`, `url_has_query`) | Mejoró v4, agotado |
| Densidad de parámetros (`content_param_density`) | Señal real pero marginal |
| Encoding Latin-1 vs ataque | Sin separación — ataques también tienen Latin-1 |
| Headers HTTP (`cookie`, `content-type`) | Sin señal — constantes en el dataset |

### Qué requeriría cerrar el gap completamente

Para separar los 936 FP de los TN se necesitaría información que las features actuales no capturan:

1. **Análisis semántico de valores de parámetros** — no solo si hay `=` o `%27`, sino qué hay en el valor de cada parámetro. Requiere parsear el body como `key=value&key=value` y analizar cada valor individualmente.
2. **Features a nivel de sesión** — ¿este request es parte de una secuencia de navegación coherente? Los ataques automatizados tienen patrones de sesión distintos.
3. **Frecuencia y ritmo** — los scanners envían requests a intervalos regulares; el tráfico normal tiene variabilidad.

Estas dimensiones requieren un enfoque diferente al feature engineering sobre campos HTTP individuales.

---

## Decisión

Las features Latin-1 tienen correlación prácticamente nula en la subpoblación relevante y dirección incorrecta en la otra. **No se incorporan al preprocessing oficial.**

### Opciones para v8

1. **Aceptar Precision 0.79 como techo práctico del enfoque actual** y avanzar a Modelo B. El MVP requiere 0.85 pero 0.79 con Recall 0.95 puede ser un punto de partida válido para producción con revisión manual de alarmas.

2. **Parseo semántico del body** — extraer los valores de los parámetros POST individualmente y construir features sobre los valores (longitud del valor, ratio de chars especiales en el valor). Mayor complejidad de implementación.

3. **Análisis del dataset CSIC 2010** — verificar si los 936 FP tienen algún patrón en campos que aún no hemos explorado (User-Agent, Accept-encoding, o el número de parámetros en la URL vs en el body).
