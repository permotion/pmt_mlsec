## Hipótesis

v5 identificó que el 51% de los FP restantes son POST sin indicadores de payload. El modelo los confunde con ataques por `content_length` y `content_param_count` altos. Feature candidata: `content_param_density` = `content_param_count / content_length` — la densidad de parámetros relativa al tamaño del body.

## Inspección del CSV crudo — root cause de los FP

Script: `notebooks/experiments/csic2010_fp_analysis_v6.py`

Se reprodujo el modelo v6 exacto (threshold=0.2903, FP=938), se identificaron los índices originales de los FP en el test set y se cruzaron con el CSV crudo para ver el contenido real.

### Hipótesis sobre headers — descartadas

Antes de la inspección se esperaba que `content-type` o `cookie` tuvieran señal:

| Header | FP | TN | Conclusión |
|---|---|---|---|
| `cookie` | 100% tienen cookie | 100% tienen cookie | ❌ No discrimina — todos los requests del dataset tienen cookie |
| `content-type` | 481 `application/x-www-form-urlencoded` + 457 NaN | 711 `application/x-www-form-urlencoded` + 3751 NaN | ❌ El NaN es simplemente GET (sin body) — no hay diferencia entre FP y TN del mismo método |

Los headers descartados en el EDA no tienen señal para este problema. La cookie es constante en todo el dataset.

### Root cause — confusión de encoding

Al inspeccionar el contenido real de los FP de **alta confianza (proba > 0.70)**, el patrón es inmediatamente obvio:

```
FP  proba=1.000  POST
content: modo=registro&login=tanchak&apellidos=Murgu%EDa+Aversa
         &email=wendell%40somosagelespa%F1a.com

FP  proba=1.000  POST
content: modo=entrar&login=clipsham5&pwd=lIMpi%24a%FA%F1as

FP  proba=0.999  POST
content: modo=registro&login=jamaly&password=exageraci%F3%3Fn
         &apellidos=Cruzado+Garmes

FP  proba=0.930  POST
content: modo=registro&apellidos=D%27Amico+Urrea

FP  proba=0.917  POST
content: modo=registro&nombre=Enrique&apellidos=Objio
         &password=escusa%F1a
```

Estos son **formularios legítimos de una tienda española** con nombres, apellidos y contraseñas que contienen caracteres acentuados encoded en ISO-8859-1/Latin-1.

### Por qué el modelo los clasifica como ataque

El modelo tiene `content_pct_density` como la segunda feature más importante (importance 959). Esta feature cuenta **todos** los `%XX` en el body, sin distinguir su tipo. Para el modelo, `%F1` (ñ) y `%27` (') producen exactamente la misma señal.

**La distinción que el modelo no puede hacer con las features actuales:**

| Tipo | Rango hex | Ejemplos concretos en los FP | Significado real |
|---|---|---|---|
| **Latin-1 inofensivo** | `%C0`–`%FF` | `%ED`=í, `%F1`=ñ, `%FA`=ú, `%F3`=ó, `%E1`=á | Vocales acentuadas españolas |
| **Latin-1 inofensivo** | `%40`–`%7E` | `%40`=@, `%24`=$, `%2B`=+ | Símbolos en emails y contraseñas |
| **Ataque** | `%00`–`%3F` | `%27`=', `%3C`=<, `%3E`=>, `%3B`=; | SQLi / XSS chars |

Los ataques reales **nunca encodean la ñ o la á** — no tienen ninguna razón para incluir caracteres Latin-1 en un payload de inyección. El tráfico legítimo de una tienda española **siempre los tiene** en nombres y contraseñas.

### El caso borde — D'Amico con %27

El apellido `D%27Amico` usa `%27` para el apóstrofe — exactamente el mismo encoding que usa SQLi para la comilla simple. Hay 5 FP con `%27` en el contenido (de 938 totales). Estos son **inevitables** con features de texto a menos que se analice el contexto completo del parámetro (el `%27` aparece en el valor de `apellidos=`, no en una construcción SQL). Son 0.5% de los FP — no mueven el needle.

### Cuantificación del problema

| Segmento de FP | Cantidad | % | Causa |
|---|---|---|---|
| Alta confianza (proba > 0.50) | 568 | 60.2% | Formularios con encoding Latin-1 denso — el modelo está muy convencido |
| Baja confianza (proba ≤ 0.50) | 375 | 39.8% | Formularios con menos encoding Latin-1 — cerca del threshold |
| Con `%27` en contenido | 8 | 0.9% | Apellidos con apóstrofe (D'Amico, O'Brien, etc.) |
| Sin ningún keyword sospechoso | 938 | **100%** | Ninguno tiene SQLi/XSS real — todos son FP estructurales |

### El gap es cerrable

La causa raíz está identificada con precisión: `content_pct_density` no distingue encoding Latin-1 de encoding de ataque. La solución es una feature que capture específicamente la proporción de Latin-1 en el body:

**Feature candidata para v7:** `content_pct_latin1_density` = proporción de secuencias `%[CcDdEeFf][0-9A-Fa-f]` en el body (código points U+00C0–U+00FF — caracteres acentuados, ñ, ü, etc.)

- **Correlación esperada:** fuertemente positiva con tráfico normal POST (formularios españoles), cercana a cero con ataques — por tres razones técnicas concretas: (1) el payload SQLi/XSS tiene que ser interpretado como código válido por el servidor, y un parser SQL no espera `ñ`; (2) los atacantes usan payloads mínimos y precisos — caracteres innecesarios solo añaden superficie de detección; (3) encoding Latin-1 puede corromper el payload si el servidor lo interpreta como ASCII o UTF-8. En el dataset CSIC 2010: **cero ataques tienen `%F1` o `%ED`**, el 100% de los requests con `%F1` son formularios legítimos.
- **Impacto esperado:** los 568 FP de alta confianza son mayoritariamente formularios con encoding Latin-1 denso → esta feature debería permitirle al modelo revertir esas predicciones
- **Equivalente para GET:** `url_pct_latin1_density` — misma lógica aplicada a la URL, para los 461 FP GET
