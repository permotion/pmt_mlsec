## Hipótesis

Las features de estructura de URL dan contexto diferente a `url_length` en la subpoblación GET. Los 935 FP GET de v3 tienen URLs más largas que el tráfico normal, pero sin indicadores de payload. La estructura del path y de la query string debería separar mejor que la longitud total.

---

## Análisis previo — subpoblación GET

Correlaciones calculadas **exclusivamente en los 43.088 requests GET** (70.6% del dataset):

| Feature | Correlación GET | Correlación global | Señal |
|---|---|---|---|
| `url_has_query` | **+0.341** | +0.175 | ✅ Ataques GET tienen query string más frecuentemente |
| `url_path_depth` | **-0.318** | -0.198 | ✅ Tráfico normal GET tiene paths más profundos |
| `url_query_length` | **+0.297** | +0.178 | ✅ Ataques GET tienen queries más largas |

Las tres features muestran correlación significativamente más fuerte en la subpoblación GET que globalmente — confirma que el análisis de subpoblación es el enfoque correcto. La correlación global está diluida por la mezcla GET+POST.

## Análisis

**Mayor salto de Precision en todo el proyecto:**
- XGBoost: 0.649 → 0.793 (+0.144) — mayor mejora en una sola iteración
- LightGBM: 0.713 → 0.803 (+0.090) — primera vez que supera 0.80
- Gap restante LightGBM: **0.803 → 0.85 = 0.047**

**Techo de ROC-AUC roto por primera vez:**
- El plateau ~0.955 que se mantuvo en v1/v2/v3 se rompió: 0.955 → **0.966**
- Las features de estructura de URL aportaron información genuinamente nueva

**FP cayeron drásticamente:**
- LightGBM: 1444 → 877 (-567) — confirmación directa de que los FP GET eran el cuello de botella
- XGBoost: 1946 → 933 (-1013) — reducción de más del 50%

**Recall 0.9492 — efecto de threshold, no de capacidad:**
- RF, XGBoost y LightGBM muestran exactamente 0.9492 — 0.0008 por debajo del criterio
- El threshold se optimiza en val; el test set da un resultado marginalmente distinto
- ROC-AUC 0.966 confirma capacidad de separación suficiente — es calibración, no información faltante
